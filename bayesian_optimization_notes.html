Bayesian Optimization — Study Notes

Algorithms · Intuition · Applied Examples · Practical Workflow

1. What is Bayesian Optimization?

BO is a strategy for finding the maximum (or minimum) of an expensive black-box function using as few evaluations as possible. "Expensive" means each evaluation costs real time, money, or both — a wet-lab experiment, a clinical test, a long simulation.

Key idea: instead of blindly testing points, build a model of the function and use that model to decide where to look next.

Contrast with:

  • Grid search — exhaustive, wasteful at high dimensions
  • Random search — no memory, no learning
  • Classical DoE (RSM) — assumes linear / quadratic relationships; breaks on nonlinear responses
  • BO — learns from every observation; balances exploration vs exploitation

2. The BO Loop — Step by Step

1
Initialize — run a small space-filling design (Sobol / Latin Hypercube). These are your "pilot" experiments. Typically 5–15 points depending on dimensionality.
2
Fit surrogate — train a Gaussian Process (GP) on all observed (x, y) pairs. The GP gives you a predicted mean μ(x) and uncertainty σ(x) everywhere in the space.
3
Optimize acquisition function — α(x) is a cheap-to-evaluate function that uses μ(x) and σ(x) to score candidate points. Find x* = argmax α(x).
4
Evaluate the true function at x*. Run the actual experiment / simulation.
5
Update — add (x*, y*) to the dataset. Go back to step 2. Repeat until budget is exhausted.

Key insight: The GP acts as a "memory" — it uses ALL past observations simultaneously to build a global picture of the function. Each new point refines this picture.

3. Gaussian Process (GP) Surrogate

A GP is a distribution over functions. Instead of fitting one curve, it fits infinitely many, weighted by how well they fit the data.

f(x) ~ GP( μ₀(x), k(x, x') )

μ₀(x) — prior mean (often set to 0 or a constant).
k(x, x') — kernel / covariance function. Encodes smoothness assumptions.

After observing data D = {(xᵢ, yᵢ)}:

Posterior mean: μₙ(x) = k(x, X)[K(X,X) + σ²ₙI]⁻¹ y
Posterior variance: σ²ₙ(x) = k(x,x) − k(x,X)[K(X,X) + σ²ₙI]⁻¹ k(X,x)
★ What this means intuitively: near observed points → σ²(x) collapses to 0 (we're confident). Far from data → σ²(x) is large (we're uncertain). BO exploits this!

Common Kernels

KernelAssumptionWhen to use
RBF / Squared ExponentialInfinitely differentiable (very smooth)Overly smooth for most real responses — rarely best
Matérn 5/2 ⭐Twice differentiable — realistic roughnessDefault for most scientific optimization problems
Matérn 3/2Once differentiableRougher/noisier responses
LinearLinear trends onlyOnly if you know the response is linear
BoTorch's SingleTaskGP defaults to Matérn 5/2. Almost always the right choice.

Hyperparameters — how does the GP set them?

The kernel has hyperparameters: lengthscale ℓ (how quickly function varies) and output scale σ_f. These are optimized by maximizing the marginal log-likelihood (MLL) of the data. BoTorch does this automatically with fit_gpytorch_mll().

4. Acquisition Functions — Deciding WHERE to Sample Next

The acquisition function α(x) takes the GP posterior and returns a score for each candidate x. We pick x* = argmax α(x) as the next experiment.

Expected Improvement (EI)
EI(x) = E[max(f(x) − f*, 0)]

f* = current best observed value.
Balances exploration & exploitation automatically.
Most widely used. Great default.

Log EI (LogEI) ⭐
LogEI(x) = log EI(x)

Numerically stable version — avoids underflow when EI values are tiny.
Use this instead of EI in practice (BoTorch default).

Upper Confidence Bound (UCB)
UCB(x) = μ(x) + β·σ(x)

β controls exploration vs exploitation.
β small → exploit (trust mean). β large → explore (seek uncertainty).
More interpretable but requires β tuning.

Probability of Improvement (PI)
PI(x) = P(f(x) > f* + ε)

ε > 0 ensures exploration isn't zero.
Greedy — tends to exploit too early.
Less popular than EI. Use only with strong prior.

The exploration–exploitation tradeoff:
— Exploration = sample where σ(x) is large (go into the unknown)
— Exploitation = sample where μ(x) is already high (refine near the known best)
EI/LogEI automatically balances both — that's why it's so robust.

5. Initialization — Sobol vs Random

Before BO starts, we need a few "cold-start" observations. The quality of initialization matters — a good spread reduces the risk of the GP being confidently wrong in unexplored regions.

★ Sobol sequences are quasi-random — they fill the space far more uniformly than pure random sampling. Use SobolEngine in PyTorch or BoTorch's built-in sampler. The gains are most visible in 4+ dimensions.

6. Applied Example #1 — Single-Objective BO

Problem: PK Database Screening → Find Top 3 Molecules

Context: You have a database of 800 candidate drug-like molecules, each with 6 computed molecular descriptors. Running a full pharmacokinetic (PK) assay on each is expensive — costs ~$2,000/compound and 5 days per batch. Goal: use BO to intelligently query the database and identify the top 3 compounds by oral bioavailability (F%) within a budget of 40 assays.

7. Multi-Objective BO (Pareto Optimization)

Real problems rarely have a single objective. Often you need to balance two or more conflicting goals simultaneously — improving one usually hurts another. The solution is to find the Pareto front: the set of points where you can't improve any objective without making at least one other worse.

Grey dots = dominated (there's a Pareto point that beats them on all objectives). Blue dots = non-dominated = the Pareto front. You choose your operating point on the front based on business priorities.

Applied Example #2 — Multi-Objective BO

Problem: Bioreactor (PT) — Maximize Yield, Minimize Impurity

Context: You're optimizing a fed-batch bioreactor for monoclonal antibody production. Each run takes 14 days and costs ~$15k in media + labor. You want to jointly: (1) maximize titer (g/L) and (2) minimize host-cell protein (HCP) impurity (ppm) — a key quality attribute. Budget: 30 runs.

Design variables

ParameterRangeRole
Temperature shift point (day)Day 3–7Controls growth vs production phase
pH setpoint6.8–7.4Enzyme activity, cell viability
Glucose feed rate (g/L/day)1.0–4.0Energy source; excess → overflow metabolism
Dissolved O₂ setpoint (%)20–60%Oxidative stress vs growth

Algorithm: qNEHVI (q-Noisy Expected Hypervolume Improvement)

Hypervolume improvement = how much does adding a new Pareto point expand the volume of space dominated by the current Pareto set? qNEHVI maximizes this expected expansion, accounting for noise in observations.
α_qNEHVI(x) = E[ HV(Pareto set ∪ {f(x)}) − HV(Pareto set) ]

Why qNEHVI over alternatives?

  • scalarization (weighted sum) — only finds convex Pareto points; misses concave regions
  • ε-constraint — converts to single-obj; requires you to pick ε upfront
  • qNEHVI — finds the whole Pareto front, handles noise, supports batch queries

What the loop looks like in code (pseudocode)

# Initialize
X = sobol_sample(n=8)   # 8 initial bioreactor runs
Y = run_bioreactor(X)   # Y has shape (8, 2): [titer, -HCP]
                        # note: negate HCP to convert to maximization

# BO loop
for step in range(22):
    model = MultiTaskGP(X, Y)           # one GP per objective
    fit_gpytorch_mll(mll)

    ref_point = Y.min(dim=0).values - 0.1   # reference point below all obs
    acqf = qNEHVI(model, ref_point=ref_point, sampler=SobolSampler(128))

    x_next = optimize_acqf(acqf, bounds, q=1)   # suggest 1 run
    y_next = run_bioreactor(x_next)
    X, Y = append(X, x_next), append(Y, y_next)

pareto_mask = is_non_dominated(Y)
pareto_front = Y[pareto_mask]

9. Practical Workflow — How to Apply BO to a New Problem

  1. Define your design space — list all input variables (continuous / categorical), their ranges, and any hard constraints (mass balance, safety limits, feasibility)
  2. Define objective(s) — single target (maximize/minimize) or multiple (Pareto). Make sure every objective is measurable with a consistent noise model
  3. Choose your budget — how many evaluations can you afford? Rule of thumb: n_init ≈ 2× dimensionality; remaining budget goes to BO steps
  4. Initialize with Sobol — not random. This spreads points well and reduces GP cold-start bias
  5. Fit GP + optimize acquisition — SingleTaskGP + LogEI for single-obj; MultiTaskGP + qNEHVI for multi-obj
  6. Run the experiment — get the real observation. Add noise estimate if possible (duplicate experiments help)
  7. Monitor convergence — plot best-so-far vs iteration. Flat curve = converged or budget exhausted
  8. Inspect the GP posterior — check uncertainty maps to see where the model is still unsure
  9. Report top candidates — include GP predicted mean ± CI, not just the point estimate

10. Common Pitfalls

  • Too few init points — GP starts with a poorly conditioned prior; misleads acquisition function early on
  • Ignoring noise — if your experiment has variability (σ > 0), use the noisy GP variant (add noise to likelihood)
  • Wrong kernel — RBF is too smooth for most real responses; use Matérn 5/2 as default
  • Unnormalized inputs — GP is sensitive to scale; always normalize inputs to [0,1] (BoTorch's Normalize does this)
  • Unnormalized outputs — standardize outputs to zero mean / unit variance (Standardize in BoTorch)
  • Batch vs sequential — if you can run experiments in parallel, use qLogEI (batch acquisition) instead of single-point EI
  • Multi-objective: bad reference point — set ref_point just below the worst observed values, not at zero. Wrong ref point distorts Pareto volume computation
Quick library cheatsheet:
• Single-obj → SingleTaskGP + LogEI
• Multi-obj → SingleTaskGP × n_obj + qNEHVI
• Batch → qLogEI / qNEHVI with q>1
• All via BoTorch
